# Data Model: Algorithm Visualizer Framework

**Feature**: 001-visualizer-framework
**Date**: 2025-10-05

## Overview

This document defines the core data entities, their attributes, relationships, and state transitions for the algorithm visualization framework.

---

## Core Entities

### 1. AlgorithmPlugin

**Purpose**: Represents a complete algorithm visualization package

**Attributes**:
| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `id` | string | ✅ | Unique identifier (kebab-case, e.g., "bubble-sort") |
| `name` | string | ✅ | Display name for UI (e.g., "Bubble Sort") |
| `description` | string | ✅ | 2-3 sentence explanation of algorithm |
| `category` | string | ✅ | Grouping category (e.g., "Sorting", "Graph") |
| `visualizationType` | enum | ✅ | Renderer type: 'grid' \| 'array' \| 'tree' \| 'graph' |
| `inputSchema` | InputSchema | ✅ | Defines expected input format for UI generation |
| `presets` | InputPreset[] | ✅ | Pre-configured example inputs (min 2) |
| `trace` | function | ✅ | `(input) => Trace` - generates execution history |
| `validateInput` | function | ✅ | `(input) => ValidationResult` - input validation |
| `code` | CodeDefinition? | ❌ | Optional source code for highlighting |

**Relationships**:
- **Has many** `InputPreset` (embedded, min 2, max 10 recommended)
- **Produces** `Trace` when trace() called with valid input
- **References** visualization component type via `visualizationType`

**Uniqueness**: `id` must be unique across all plugins

**Validation Rules**:
- `id` must match pattern: `/^[a-z0-9-]+$/`
- `presets` must contain at least 2 items (tiny/simple + classic)
- `trace()` must return frames with sequential step numbers starting at 0
- `validateInput()` must return `{ valid: true }` for all preset inputs

**Lifecycle**:
1. **Registration**: Plugin discovered via `import.meta.glob` at build time
2. **Validation**: Framework validates plugin implements `AlgorithmPlugin` interface
3. **Catalog**: Plugin added to algorithm selector UI
4. **Active**: Plugin selected by user, becomes active algorithm

### 2. Frame

**Purpose**: Immutable snapshot of algorithm state at a single execution step

**Attributes**:
| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `step` | number | ✅ | Sequential step number (0-indexed) |
| `state` | any | ✅ | Algorithm-specific state data (grid, tree, array, etc.) |
| `focus` | FocusMarker[]? | ❌ | Cells/nodes being processed (amber highlight) |
| `neighbors` | FocusMarker[]? | ❌ | Related elements (green highlight) |
| `metrics` | Record<string, any>? | ❌ | Algorithm metrics (e.g., `{ totalWater: 42 }`) |
| `description` | string | ✅ | Human-readable explanation of step action |

**Relationships**:
- **Belongs to** exactly one `Trace`
- **Embedded** `FocusMarker` instances (no separate persistence)

**Uniqueness**: `step` is unique within a Trace (sequential, no gaps)

**Validation Rules**:
- `step` must be >= 0 and sequential (no skipped numbers)
- `state` structure must match plugin's state type
- `description` must be non-empty string (min 10 chars recommended)
- `focus` and `neighbors` IDs must reference valid elements in `state`

**Immutability**: Frames MUST NOT be mutated after creation (constitutional requirement III)

**Lifecycle**:
1. **Creation**: Generated by plugin's `trace()` function
2. **Sampling**: Optionally sampled if total frames > 5000 (FR-004a)
3. **Storage**: Stored in `Trace.frames` array
4. **Rendering**: Read by PlaybackController, passed to visualization components

---

### 3. Trace

**Purpose**: Complete execution history for an algorithm run with specific input

**Attributes**:
| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `frames` | Frame[] | ✅ | Sequential array of execution frames |
| `totalSteps` | number | ✅ | Total number of steps (`frames.length`) |
| `completed` | boolean | ✅ | Whether algorithm finished successfully |
| `metadata` | Record<string, any>? | ❌ | Execution metadata (time, memory, etc.) |

**Relationships**:
- **Has many** `Frame` (sequential, immutable collection)
- **Produced by** one `AlgorithmPlugin.trace()` call
- **Consumed by** one `PlaybackController` instance

**Uniqueness**: Trace is ephemeral (regenerated on new input, not persisted)

**Validation Rules**:
- `frames` must not be empty (min 2: initial + final state)
- `frames[i].step` must equal `i` (sequential validation)
- `totalSteps` must equal `frames.length`
- If `completed === false`, metadata should explain why

**State Transitions**:
```
[Input Provided] → [Validation] → [Trace Generation] → [Optional Sampling] → [Playback]
                       ↓ invalid                             ↓ >5k frames
                   [Error State]                          [Sampled Trace]
```

**Lifecycle**:
1. **Generation**: User submits input → plugin validates → plugin.trace() called
2. **Sampling**: If `frames.length > 5000`, apply adaptive stride sampling
3. **Active**: Trace becomes active in PlaybackController
4. **Disposal**: New input submitted → old trace discarded, new trace generated

---

### 4. FocusMarker

**Purpose**: Identifies a specific element for visual highlighting during a frame

**Attributes**:
| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `type` | enum | ✅ | Element type: 'grid-cell' \| 'array-index' \| 'tree-node' \| 'graph-node' |
| `id` | string | ✅ | Unique element identifier (e.g., "2,3" for grid cell) |
| `style` | Record<string, any>? | ❌ | Optional styling hints (color, intensity, etc.) |

**Relationships**:
- **Embedded in** `Frame.focus` or `Frame.neighbors` arrays
- **References** element in `Frame.state` (validation required)

**Uniqueness**: `id` must reference a valid element in the frame's state

**ID Format by Type**:
- **grid-cell**: `"row,col"` (e.g., "2,3" for row 2, col 3)
- **array-index**: `"index"` (e.g., "5" for arr[5])
- **tree-node**: `"node-{id}"` (e.g., "node-7" for node with ID 7)
- **graph-node**: `"node-{id}"` (same as tree)

**Validation Rules**:
- `type` must match parent plugin's `visualizationType`
- `id` format must match type conventions
- Referenced element must exist in `Frame.state`

### 5. PlaybackController

**Purpose**: Manages visualization playback state and user controls

**Attributes** (Svelte 5 Runes):
| Attribute | Type | Reactive | Description |
|-----------|------|----------|-------------|
| `currentIndex` | number | `$state` | Current frame index (0 to trace.totalSteps - 1) |
| `isPlaying` | boolean | `$state` | Play/pause status |
| `speed` | number | `$state` | Playback delay in ms (50-2000) |
| `trace` | Trace? | `$state.raw` | Active trace (non-reactive to prevent deep watching) |
| `currentFrame` | Frame? | `$derived` | Computed from trace.frames[currentIndex] |
| `progress` | number | `$derived` | Computed percentage (currentIndex / totalSteps * 100) |

**Relationships**:
- **Manages** one active `Trace`
- **Provides** current `Frame` to visualization components
- **Controls** animation timer via `$effect`

**State Transitions**:
```
[Idle] ←→ [Playing]
  ↓         ↑ ↓
[Step Forward/Back]
  ↓
[Frame Change] → [Component Re-render]
```

**Methods**:
- `play()`: Start automatic playback
- `pause()`: Stop playback, preserve currentIndex
- `stepForward()`: Increment currentIndex by 1 (capped at totalSteps - 1)
- `stepBack()`: Decrement currentIndex by 1 (capped at 0)
- `reset()`: Set currentIndex to 0
- `setSpeed(ms: number)`: Update playback delay
- `loadTrace(trace: Trace)`: Replace active trace, reset to step 0

**Lifecycle**:
1. **Initialization**: Created when algorithm selected
2. **Trace Loading**: User submits input → trace generated → loadTrace() called
3. **Playback Loop**: If isPlaying, $effect increments currentIndex every `speed` ms
4. **Disposal**: Component unmount clears interval via $effect cleanup

### 6. InputPreset

**Purpose**: Pre-configured example input for quick experimentation

**Attributes**:
| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `name` | string | ✅ | Display name (e.g., "Tiny 3×3 bowl") |
| `data` | any | ✅ | Actual input data (must pass plugin validation) |
| `description` | string? | ❌ | Brief explanation shown in UI tooltip |

**Relationships**:
- **Belongs to** one `AlgorithmPlugin` (embedded in `presets` array)

**Validation Rules**:
- `data` must pass parent plugin's `validateInput()` function
- `name` must be unique within plugin's presets array
- Recommended categories: "Tiny" (simple), "Classic" (typical), "Complex" (edge case)

**Lifecycle**: Static, defined at plugin creation time

---

### 7. CodeDefinition (Optional)

**Purpose**: Source code for synchronized highlighting during visualization

**Attributes**:
| Attribute | Type | Required | Description |
|-----------|------|----------|-------------|
| `language` | string | ✅ | Programming language ('typescript', 'python', 'pseudocode') |
| `source` | string | ✅ | Full source code with line breaks |
| `stepToLines` | Record<number, [number, number]>? | ❌ | Map step → highlighted line range |

**Relationships**:
- **Belongs to** one `AlgorithmPlugin` (optional field)
- **Consumed by** Monaco Editor component for highlighting

**Validation Rules**:
- `language` must be Monaco-supported language ID
- `stepToLines` keys must be valid step numbers from trace
- Line numbers in ranges must be >= 1 and <= total source lines

**Fallback Behavior** (FR-019a):
- If `code` is undefined, framework auto-generates pseudocode from `Frame.description` text
- Generated pseudocode uses step descriptions as line-by-line statements
- Auto-highlighting maps step N to line N in generated pseudocode

**Lifecycle**: Static, defined at plugin creation time (if provided)

## State Type Definitions (TypeScript)

### Grid State (for grid-based algorithms)
```typescript
interface GridState {
  grid: number[][];           // Current grid values
  visited: boolean[][];        // Which cells have been processed
  water?: number[][];          // Optional overlays (e.g., water levels)
  heap?: [number, number, number][]; // Optional data structures (e.g., min-heap)
  [key: string]: any;          // Algorithm-specific extensions
}
```

### Array State (for sorting/searching algorithms)
```typescript
interface ArrayState {
  array: number[];             // Current array values
  sorted?: boolean[];          // Which elements are in final position
  comparisons?: number;        // Algorithm metrics
  swaps?: number;
  [key: string]: any;
}
```

### Tree State (for tree algorithms)
```typescript
interface TreeNode {
  id: string;
  value: any;
  children: string[];          // IDs of child nodes
  parent?: string;
}

interface TreeState {
  nodes: Record<string, TreeNode>; // Node ID -> Node data
  root: string;                    // Root node ID
  [key: string]: any;
}
```

### Graph State (for graph algorithms)
```typescript
interface GraphNode {
  id: string;
  value: any;
  x?: number;                   // Layout coordinates
  y?: number;
}

interface GraphEdge {
  from: string;                 // Node ID
  to: string;
  weight?: number;
}

interface GraphState {
  nodes: Record<string, GraphNode>;
  edges: GraphEdge[];
  [key: string]: any;
}
```

## Validation Rules Summary

| Entity | Key Validations |
|--------|----------------|
| **AlgorithmPlugin** | Unique ID, min 2 presets, trace returns sequential frames |
| **Frame** | Sequential step, non-empty description, valid focus IDs |
| **Trace** | Min 2 frames, totalSteps matches length, sequential steps |
| **FocusMarker** | Valid ID format, references existing state element |
| **PlaybackController** | currentIndex within bounds [0, totalSteps-1] |
| **InputPreset** | Data passes plugin validation, unique name per plugin |
| **CodeDefinition** | Valid language, stepToLines within source bounds |

## Performance Considerations

1. **Trace Storage**: Use `$state.raw()` for frame arrays to prevent deep reactivity (Svelte 5)
2. **Smart Sampling**: Traces >5000 frames auto-sampled during generation (not reactively)
3. **Frame Immutability**: Prevents accidental mutations, enables efficient change detection
4. **Focus Markers**: Max 20 focus markers per frame recommended (DOM rendering limit)
5. **State Structure**: Keep state objects shallow where possible (faster cloning during trace)
